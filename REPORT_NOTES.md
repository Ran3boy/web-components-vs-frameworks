# Заметки для отчета ВКР

*Данный документ содержит текстовые заготовки, которые можно использовать при написании пояснительной записки или отчета по практике.*

### 1. Описание практического решения
В качестве практического решения был разработан интерактивный стенд на базе Vite, TypeScript и React. Приложение демонстрирует одновременную реализацию идентичного UI-компонента (карточки товара) двумя способами: с использованием компонентного подхода фреймворка React и с использованием нативной технологии Web Components (Custom Elements + Shadow DOM). 

### 2. Обоснование выбора развертывания
Для унификации среды запуска проект был контейнеризирован с использованием Docker. Использование Docker-образа исключает проблему "работает на моем компьютере" и позволяет руководителю или проверяющему запустить проект одной командой `docker compose up`. Раздача статического контента внутри контейнера осуществляется легковесным веб-сервером Nginx, что является индустриальным стандартом.

### 3. Описание Docker-образа
Для проекта настроен Multi-stage сборка (многоэтапный `Dockerfile`). 
- На **первом этапе** (базовый образ `node:20-alpine`) происходит установка зависимостей и сборка приложения (компиляция TypeScript в JavaScript). 
- На **втором этапе** (базовый образ `nginx:alpine`) сгенерированные статические файлы копируются в директорию сервера Nginx. 
Такой подход позволил минимизировать итоговый размер Docker-образа, так как в него не попадают исходные коды и тяжелая папка `node_modules`.

### 4. Описание CI/CD-конвейера
Для автоматизации проверки и доставки кода настроен CI/CD pipeline с использованием GitHub Actions. Конвейер состоит из двух основных задач (`jobs`): `build` и `deploy`. При отправке изменений в главную ветку репозитория конвейер автоматически разворачивает изолированное окружение, проверяет код, собирает статические файлы и публикует обновленное приложение на GitHub Pages, делая его доступным по публичной ссылке.

### 5. Выводы по сравнению Web Components и Frontend-фреймворков
На основе разработанного стенда можно сделать следующие выводы:
1. **Web Components** обеспечивают превосходную, недостижимую для фреймворков изоляцию стилей благодаря `Shadow DOM`. Они не зависят от сторонних библиотек, что делает их идеальным выбором для создания переиспользуемых UI-китов (Design Systems), которые можно внедрять в любые другие проекты независимо от стека технологий. Сборка компонента занимает 0 Кб дополнительного веса (используется API браузера).
2. **Сложность разработки** на нативном API Web Components выше, чем в современных фреймворках, так как разработчику приходится вручную управлять подписками на атрибуты (`attributeChangedCallback`) и обновлять DOM.
3. Фреймворки, такие как **React**, предлагают более удобный декларативный подход (JSX), упрощают управление состоянием и реактивность. Однако их использование добавляет зависимость (около 40 Кб базового веса React) и привязывает компоненты к экосистеме фреймворка.
4. **Итог:** Выбор технологии зависит от задачи. Для микрофронтендов и универсальных UI-библиотек лучше подходят Web Components. Для быстрого создания сложных бизнес-приложений — современные фронтенд-фреймворки.
