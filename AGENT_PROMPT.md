## Требования к Docker

Создай `Dockerfile`, который выполняет:

1. установку зависимостей;
2. сборку frontend-приложения;
3. запуск production-версии через nginx.

Примерная логика Dockerfile:

```Dockerfile
FROM node:20-alpine AS build

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .
RUN npm run build

FROM nginx:alpine

COPY --from=build /app/dist /usr/share/nginx/html

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```

Также создай `docker-compose.yml`:

```yaml
services:
  app:
    build: .
    ports:
      - "8080:80"
```

## GitHub Actions

Создай файл:

```text
.github/workflows/ci-cd.yml
```

Pipeline должен выполнять:

- загрузку репозитория;
- установку Node.js;
- установку зависимостей;
- запуск тестов;
- сборку проекта;
- публикацию приложения на GitHub Pages.

Пример структуры pipeline:

```yaml
name: CI/CD

on:
  push:
    branches:
      - main

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test -- --run
        continue-on-error: true

      - name: Build project
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: dist

  deploy:
    runs-on: ubuntu-latest
    needs: build

    environment:
      name: github-pages

    steps:
      - name: Deploy to GitHub Pages
        uses: actions/deploy-pages@v4
```

## GitLab CI/CD

Если используется GitLab, создай файл:

```text
.gitlab-ci.yml
```

Pipeline должен выполнять:

- установку зависимостей;
- тестирование;
- сборку;
- публикацию на GitLab Pages.

Пример:

```yaml
image: node:20-alpine

stages:
  - test
  - build
  - deploy

test:
  stage: test
  script:
    - npm ci
    - npm test -- --run
  allow_failure: true

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist

pages:
  stage: deploy
  script:
    - npm ci
    - npm run build
    - rm -rf public
    - mv dist public
  artifacts:
    paths:
      - public
  only:
    - main
```

## Логика расчёта итоговой оценки

Добавь функцию расчёта итоговой оценки технологии.

Создай файл:

```text
src/utils/calculateScore.ts
```

Пример логики:

```ts
type BenchmarkItem = {
  loadTimeMs: number
  renderTimeMs: number
  bundleSizeKb: number
  dependenciesCount: number
  complexityScore: number
  reusabilityScore: number
  maintainabilityScore: number
}

export function calculateFinalScore(item: BenchmarkItem): number {
  const performanceScore =
    10 - Math.min(10, (item.loadTimeMs + item.renderTimeMs) / 50)

  const bundleScore =
    10 - Math.min(10, item.bundleSizeKb / 20)

  const dependencyScore =
    10 - Math.min(10, item.dependenciesCount * 1.5)

  const simplicityScore =
    10 - item.complexityScore

  const qualityScore =
    (item.reusabilityScore + item.maintainabilityScore) / 2

  const finalScore =
    performanceScore * 0.25 +
    bundleScore * 0.2 +
    dependencyScore * 0.15 +
    simplicityScore * 0.15 +
    qualityScore * 0.25

  return Number(finalScore.toFixed(1))
}
```

В документации объясни, что итоговая оценка является сравнительной и зависит от выбранных весов критериев.

## Измерение загрузки компонентов

Добавь простой механизм измерения времени отображения компонента.

Пример:

```ts
export async function measureRenderTime(
  name: string,
  callback: () => void
): Promise<number> {
  const start = performance.now()

  callback()

  await new Promise((resolve) => requestAnimationFrame(resolve))

  const end = performance.now()

  const result = Number((end - start).toFixed(2))

  console.log(`${name} render time: ${result} ms`)

  return result
}
```

Используй эту функцию для сравнения демо-компонентов, если это возможно в выбранной архитектуре.
